# Task 3 — Agentic Financial Research Workflow

## Overview

This task implements an observable multi-agent financial research workflow focused on practical AI systems engineering.

The workflow combines:
- financial market analysis
- financial news retrieval
- LLM-based sentiment analysis
- role-based agent orchestration
- critique and review stages
- persistent workflow memory
- and execution trace logging.

One of the main goals of this task was building an agentic workflow that remains:
- explainable
- observable
- reproducible
- and easy to debug.

Instead of relying on highly autonomous black-box agent systems, the architecture intentionally favors explicit orchestration and structured communication between components.

---

# Workflow Architecture

```text
User Ticker Input
        ↓
Financial Tools Layer
- get_price_data()
- calculate_volatility()
- get_news()
- llm_sentiment()
- web_search()
        ↓
Quantitative Analyst Agent
        ↓
Sentiment Research Agent
        ↓
Risk Review Agent
        ↓
Portfolio Strategist Agent
        ↓
Structured Final Report
        ↓
Persistent Memory Cache
        ↓
Follow-Up Question Handling
```

---

# Key Features

## Structured Outputs

All major workflow stages use Pydantic schemas for validation.

This helps ensure:
- predictable outputs
- cleaner downstream orchestration
- safer workflow composition
- and improved debugging reliability.

---

## Observability and Tracing

The workflow includes JSONL-based execution tracing.

Each tool call records:
- timestamps
- inputs
- outputs
- execution duration
- and failures.

This creates transparent execution logs that make debugging and workflow inspection significantly easier.

---

## Multi-Agent Orchestration

The system separates responsibilities across focused agents:
- quantitative analysis
- sentiment research
- risk critique
- and portfolio strategy synthesis.

This keeps the workflow modular and easier to reason about.

---

## Persistent Workflow Memory

Generated reports are cached locally and reused for follow-up questions.

This demonstrates:
- practical state management
- reduced repeated API usage
- and lightweight memory-based reasoning.

---

## Graceful Degradation

The workflow intentionally handles:
- API failures
- malformed LLM responses
- missing news
- rate limits
- and partial workflow failures.

Fallback responses are structured and traceable instead of silently failing.

# Engineering Notes

One of the primary design goals was avoiding unnecessary autonomous complexity.

The workflow intentionally favors:
- explicit orchestration
- deterministic flow control
- structured validation
- modular tooling
- and observable execution.

This makes the system:
- easier to debug
- easier to explain
- easier to extend
- and more operationally realistic.

The architecture is intentionally designed to resemble a practical production-oriented AI workflow rather than a research-style autonomous agent experiment.

# Limitations

- Free-tier APIs may occasionally rate limit requests.
- Financial news quality depends on external providers.
- LLM outputs may vary slightly between runs.
- The workflow is educational and engineering-focused rather than investment-grade financial advice.
- Long-running workflows may still require retry handling in production environments.
